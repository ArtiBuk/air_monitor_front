# Air Monitor Lab — инструкция по работе с интерфейсом

Этот файл описывает, что есть в интерфейсе, какие поля за что отвечают и в каком порядке обычно работать с системой.

## Логика работы сервиса

Обычный сценарий такой:

1. Собрать наблюдения.
2. Построить срез датасета.
3. Обучить модель.
4. Сгенерировать прогноз.
5. При необходимости запустить ретропроверку.
6. Когда появятся фактические наблюдения, оценить прогноз по факту.
7. Для сравнений собрать несколько запусков в серию экспериментов.

## Что видно в шапке

- `Сессия` — имя или email текущего пользователя.
- `N задач в фокусе` — сколько фоновых задач отслеживается в этом браузере.
- `Инструкция` — скачивает этот файл.
- `Выйти` — завершает сессию.

## Страница входа

### Режим `Вход`

- `Электронная почта` — email пользователя.
- `Пароль` — пароль для входа.

### Режим `Регистрация`

- `Имя` — имя пользователя.
- `Фамилия` — фамилия пользователя.
- `Электронная почта` — логин и основной идентификатор.
- `Пароль` — пароль нового пользователя.

## Раздел `Обзор`

Главный экран даёт быстрый срез по системе.

### Верхние карточки

- `Наблюдения` — сколько последних записей вернул API.
- `Датасеты` — сколько последних срезов датасета доступно.
- `Модели` — сколько последних версий модели найдено.
- `Прогнозы` — сколько последних запусков прогноза найдено.
- `Эксперименты` — сколько последних исследовательских запусков найдено.
- `Серии` — сколько последних исследовательских серий найдено.

### Блок `Последний прогноз`

- Показывает график последнего прогноза.
- Если прогнозов ещё нет, это нормально: сначала нужно обучить модель и запустить прогноз.

### Блок `Активная модель`

- Показывает модель, которую backend считает основной по умолчанию.
- Основные метрики:
  - `Общая RMSE`
  - `Общая MAE`
  - `Макро MAPE`
- `Цели` — какие метрики эта модель предсказывает.

### Блок `Лидерборд моделей`

- Это агрегированная таблица по уже оценённым прогнозам.
- Пока прогнозы не оценивались по факту или через backtest, лидерборд может быть пустым.

## Раздел `Наблюдения`

Этот экран нужен для просмотра последних данных и для ручного запуска сбора.

### Поля фильтрации

- `Метрика` — фильтр по названию метрики, например `pm25`.
- `Источник` — фильтр по провайдеру, например `mycityair`.
- `Лимит` — сколько строк загрузить в таблицу.

### Поля формы сбора

- `Интервал`
  - шаг агрегации;
  - обычно используется `1 час`.
- `Старт окна`
  - начало периода, за который backend будет собирать данные.
- `Финиш окна`
  - конец периода сбора.
- `Размер окна, ч`
  - размер одного окна запросов к провайдеру;
  - обычно оставляют `1`.
- `Запланировать на`
  - необязательное поле;
  - если заполнить, задача будет создана как отложенная.

### Кнопки

- `Синхронно собрать`
  - выполняет сбор сразу и ждёт результат в этом же запросе.
- `Поставить в очередь`
  - отправляет задачу в Celery;
  - удобно для более длинных запусков.

### Что значат результаты сбора

- `Получено` — сколько сырых записей пришло из источника.
- `Очищено` — сколько записей прошло очистку и нормализацию.
- `Создано` — сколько новых записей добавлено в БД.
- `Обновлено` — сколько существующих записей было обновлено.

### Практическая рекомендация

Если стенд новый, сначала лучше собрать не 24 часа, а хотя бы 5-7 суток истории. Иначе типичный датасет с окнами `72/24` не соберётся.

## Раздел `Датасеты`

Здесь строятся срезы датасета для обучения.

### Поля формы

- `Входное окно, ч`
  - сколько часов истории подавать в модель на вход.
- `Горизонт прогноза, ч`
  - на сколько часов вперёд строится прогноз.
- `Признаки`
  - список входных колонок через запятую.
- `Целевые метрики`
  - список колонок, которые модель должна предсказывать.
- `Запланировать на`
  - отложенный запуск сборки датасета.

### Кнопки

- `Собрать синхронно`
- `Собрать асинхронно`

### Что важно понимать

- Если признаки и цели не заполнены, backend может взять значения по умолчанию.
- Форма старается подставить параметры последнего датасета автоматически.
- Если `Примеров` в результате мало или ноль, значит для выбранных окон пока не хватает наблюдений.

### Что видно в таблице

- `ID` — идентификатор среза.
- `Окно` — `input_len_hours`.
- `Горизонт` — `forecast_horizon_hours`.
- `Строк` — размер мастер-таблицы до формирования sample.
- `Примеров` — сколько обучающих sample удалось собрать.
- `Признаков` — количество feature columns.
- `Целей` — количество target columns.
- `Создан` — время создания.

## Раздел `Модели`

Этот экран отвечает за обучение и просмотр результатов.

### Поля формы обучения

- `Срез датасета`
  - конкретный `DatasetSnapshot`, на котором будет train;
  - если не выбран, backend пытается взять актуальный подходящий срез.
- `Эпохи`
  - сколько проходов сделать по обучающим данным.
- `Размер батча`
  - размер батча при обучении.
- `Скорость обучения`
  - `learning rate`.
- `Регуляризация`
  - `weight decay`.
- `Терпение early stop`
  - сколько неудачных эпох ждать перед ранней остановкой.
- `Случайное зерно`
  - seed для воспроизводимости.
- `Запланировать на`
  - отложенный запуск обучения.

### Кнопки

- `Обучить синхронно`
- `Запустить через Celery`

### Что видно рядом

- `Активная модель`
  - текущая базовая модель по умолчанию.
- `Версии моделей`
  - последние обученные версии.
- `Лидерборд по оценке`
  - сводная статистика по результатам backtest и оценки по факту.

### Статусы моделей

- `ready` — модель готова к прогнозам.
- `training` — обучение ещё идёт.
- `failed` — обучение завершилось ошибкой.

Для генерации прогноза используются только модели со статусом `ready`.

## Раздел `Прогнозы`

Здесь строятся обычные прогнозы, запускается ретропроверка и выполняется оценка по факту.

### Поля формы

- `Версия модели`
  - модель для прогноза;
  - если не выбрана, используется активная модель;
  - в списке показываются только `ready` модели.
- `Входное окно, ч`
  - сколько часов истории брать перед прогнозом.
- `Горизонт прогноза, ч`
  - длина прогноза вперёд.
- `Дата отсечения для backtest`
  - нужна только для исторической ретропроверки;
  - это точка, из которой прогноз строится как будто "в прошлом".
- `Запланировать на`
  - работает только для фоновой генерации обычного прогноза.

### Кнопки

- `Построить сразу`
  - обычный прогноз прямо сейчас.
- `Запустить в фоне`
  - обычный прогноз через Celery.
- `Запустить ретропроверку`
  - прогноз на исторической точке отсечения.

### Блок `Выбранный прогноз`

- показывает график и краткую сводку по выбранному запуску;
- если оценка уже выполнена, появятся `Покрытие`, `RMSE`, `MAE`;
- если фактические данные ещё не дошли до конца горизонта, кнопка `Оценить по факту` будет недоступна.

### Что значит оценка по факту

- backend сравнивает прогноз с реальными наблюдениями на тех же временных точках;
- если нужных наблюдений ещё нет, оценка не выполняется;
- если прошлый запуск оценки упал, его можно повторить позже.

## Раздел `Эксперименты`

Это экран для исследовательских прогонов.

### Блок `Создать series`

- `Название` — короткое имя серии.
- `Цель` — краткое описание гипотезы или идеи сравнения.
- `Описание` — более подробное пояснение.

Серия нужна, чтобы объединять несколько запусков в одну исследовательскую группу.

### Блок `Запуск эксперимента`

#### Общие поля

- `Название эксперимента` — имя конкретного запуска.
- `Серия` — к какой серии привязать запуск.

#### Параметры датасета

- `Входное окно, ч`
- `Горизонт прогноза, ч`
- `Признаки`
- `Целевые метрики`

#### Параметры обучения

- `Эпохи`
- `Размер батча`
- `Случайное зерно`

#### Дополнительные поля

- `Дата отсечения для backtest`
  - если заполнена, эксперимент дополнительно выполнит ретропроверку.
- `Запланировать на`
  - отложенный запуск всего сценария.

### Что делает эксперимент

Один `ExperimentRun` может включать:

- сборку датасета;
- обучение модели;
- генерацию прогноза;
- ретропроверку;
- оценку и итоговую сводку.

### Блок `Отчёт по серии`

Показывает агрегаты по выбранной серии:

- сколько запусков было;
- сколько завершено;
- среднюю `RMSE` обучения;
- среднюю `RMSE` ретропроверки.

## Раздел `Задачи`

Этот экран нужен для фоновых и отложенных задач.

### Блок `Добавить внешний ID задачи`

- нужен, если задача была создана не из этого браузера;
- сюда можно вручную вставить `task_id` и начать отслеживание.

### Блок `Отслеживаемые задачи`

- хранится в `localStorage`;
- задачи не пропадают при переходе между страницами;
- кнопку `Убрать` можно использовать, чтобы очистить локальный список.

### Блок `Отложенные задачи`

Показывает задачи, у которых был заполнен `scheduled_for`.

Поля:

- `Операция`
- `Статус`
- `Запланирована`
- `Начата`
- `Завершена`

Если статус ещё `scheduled`, такую задачу можно отменить.

## Общие обозначения

### Что такое `sync` и `async`

- `sync` — запрос выполняется сразу, браузер ждёт ответ.
- `async` — операция ставится в Celery и продолжается в фоне.

### Что такое `scheduled_for`

Это время, на которое нужно отложить запуск задачи. Если поле пустое, задача стартует сразу.

### Почему ID в таблицах короткие

Во многих таблицах интерфейс показывает только первые 8 символов UUID, чтобы таблица читалась проще. Полный ID обычно доступен в raw-ответе или на backend.

### Что означает raw в карточках результата

После успешной операции интерфейс показывает краткую сводку. При необходимости он также хранит полный ответ backend, чтобы можно было посмотреть исходный payload результата.

## Практические советы

- На пустом стенде сначала собери несколько суток наблюдений.
- Не пытайся строить прогноз без готовой модели.
- Не сравнивай модели по одной train-метрике; лучше ориентироваться на backtest и оценку по факту.
- Для повторяемых сравнений создавай `series`, а не отдельные несвязанные запуски.
- Если долгие операции выполняются в фоне, контролируй их через раздел `Задачи`.
