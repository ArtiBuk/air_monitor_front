# Air Monitor Lab — инструкция пользователя

Этот файл описывает, как работать с интерфейсом мониторинга и прогнозирования качества воздуха.

## Основной сценарий

Обычный порядок работы такой:

1. Собрать наблюдения.
2. Собрать датасет.
3. Обучить модель.
4. Построить прогноз.
5. При необходимости запустить ретропроверку.
6. Когда появятся фактические данные, выполнить оценку прогноза.
7. Для сравнений использовать эксперименты и серии.

## Что видно в шапке

- `Сессия` — текущий пользователь.
- `N задач в фокусе` — сколько фоновых задач отслеживается в этом браузере.
- `Инструкция` — скачивает этот файл.
- `Выйти` — завершает сессию.

## Общие обозначения

- `sync` — операция выполняется сразу, браузер ждёт ответ.
- `async` — операция уходит в Celery и продолжается в фоне.
- `scheduled_for` — отложенный запуск. Если поле пустое, задача стартует сразу.
- Короткие ID в таблицах нужны для компактного отображения. Полные значения остаются в backend.

## Цветовые статусы показателей

В интерфейсе показатели могут подсвечиваться по уровню риска:

- `Норма` — мягкий зелёный.
- `Выше нормы` — мягкий жёлтый.
- `Критический` — мягкий красный.

Такая подсветка используется в карточках прогноза, AQI-блоке и таблице наблюдений.

## Раздел `Обзор`

Это главный экран состояния системы.

### Верхние карточки

Показывают общее количество:

- наблюдений;
- датасетов;
- моделей;
- прогнозов;
- экспериментов;
- серий.

### Блок `AQI`

Это ключевой показатель интерфейса.

Здесь показываются:

- последнее прогнозное значение `AQI`;
- категория риска;
- время последней точки прогноза в человекочитаемом виде;
- доминирующий риск среди метрик;
- входит ли `AQI` в цели активной модели.

### Блок `Последний прогноз`

- показывает карточки и графики по последнему успешному прогнозу;
- метрики выводятся русскими названиями;
- значения помечаются цветовым статусом.

### Блок `Активная модель`

Показывает:

- статус модели;
- название;
- сводные метрики качества;
- список целевых показателей.

### Блок `Автосбор наблюдений`

Показывает параметры hourly-задачи Celery:

- запуск по расписанию;
- окно, из которого забираются данные;
- интервал агрегации;
- размер окна;
- активные источники.

На текущем стенде hourly-сбор настроен так:

- запускается каждый час в `:05`;
- забирает последние `48` часов;
- использует `Interval1H`;
- размер окна запроса — `1` час.

## Раздел `Наблюдения`

Здесь можно вручную запускать сбор и смотреть последние данные.

### Фильтры

- `Метрика` — фильтр по показателю.
- `Источник` — фильтр по провайдеру.
- `Лимит` — сколько строк показать.

### Поля формы сбора

- `Интервал` — шаг агрегации, обычно `1 час`.
- `Старт окна` — начало периода сбора.
- `Финиш окна` — конец периода.
- `Размер окна, ч` — размер одного запроса к источнику.
- `Запланировать на` — необязательный отложенный запуск.

### Кнопки

- `Синхронно собрать` — выполнить сразу.
- `Поставить в очередь` — отправить в Celery.

### Таблица наблюдений

В таблице показываются:

- время;
- источник;
- станция;
- русское название метрики;
- значение и его цветовой статус;
- единица измерения;
- тип записи.

### Практическая рекомендация

На пустом стенде сначала лучше собрать не 24 часа, а хотя бы 5-7 суток истории. Иначе датасет с типичными окнами `72/24` может не собраться.

## Раздел `Датасеты`

Здесь собираются срезы данных для обучения.

### Поля формы

- `Входное окно, ч`
- `Горизонт прогноза, ч`
- `Признаки`
- `Целевые метрики`
- `Запланировать на`

### Что важно

- если признаки и цели не заполнены, backend может подставить значения по умолчанию;
- форма старается использовать параметры последнего датасета;
- если примеров мало или ноль, значит пока не хватает наблюдений под выбранные окна.

## Раздел `Модели`

Этот экран нужен для обучения и выбора рабочей модели.

### Поля формы

- `Срез датасета`
- `Эпохи`
- `Размер батча`
- `Скорость обучения`
- `Регуляризация`
- `Терпение early stop`
- `Случайное зерно`
- `Запланировать на`

### Что важно

- для прогноза используются только модели со статусом `ready`;
- интерфейс показывает активную модель и её целевые метрики;
- если среди целей есть `AQI`, это отдельно отражается на экране `Обзор`.

## Раздел `Прогнозы`

Здесь создаются обычные прогнозы, запускается backtest и выполняется оценка по факту.

### Поля формы

- `Версия модели`
- `Входное окно, ч`
- `Горизонт прогноза, ч`
- `Дата отсечения для backtest`
- `Запланировать на`

### Кнопки

- `Построить сразу`
- `Запустить в фоне`
- `Запустить ретропроверку`

### Что показывает экран

- выбранный запуск прогноза;
- графики по метрикам;
- сводку по оценке, если она уже выполнена;
- доступность кнопки оценки по факту.

Оценка по факту возможна только тогда, когда в системе уже есть реальные наблюдения на весь горизонт прогноза.

## Раздел `Эксперименты`

Экран упрощён и рассчитан на быстрый запуск без перегруза настройками.

### Основной сценарий

1. Выбрать режим запуска.
2. Проверить окно и горизонт.
3. Запустить эксперимент сразу или в фоне.

### Пресеты

- `Быстрый` — короткая проверка гипотезы.
- `Сбалансированный` — основной рабочий режим.
- `Исследовательский` — более тяжёлый прогон.

### Что видно на странице

- краткая инструкция в 3 шага;
- блок быстрого запуска;
- сводка по текущему запуску;
- расширенные настройки, если нужны тонкие параметры;
- отчёт по выбранной серии.

## Раздел `Задачи`

Экран для фоновых и отложенных задач.

### Что можно делать

- добавить внешний `task_id`, если задача была создана не из этого браузера;
- отслеживать текущие задачи;
- убирать задачи из локального списка;
- смотреть отложенные задачи и отменять их, пока они ещё в статусе `scheduled`.

## Практические советы

- сначала собирай историю наблюдений, потом уже датасеты и модели;
- не строй прогноз без готовой модели;
- смотри не только на train-метрики, но и на backtest и оценку по факту;
- для повторяемых сравнений используй серии экспериментов;
- если запускаешь долгие операции в фоне, контролируй их через раздел `Задачи`.
