# air_monitor_front

Frontend для `air_monitor_back`: интерфейс управления пайплайном мониторинга и краткосрочного прогнозирования качества воздуха в Норильском промышленном районе.

Приложение закрывает прикладной сценарий целиком: от авторизации и ручного запуска операций до карты качества воздуха, контроля фоновых задач и скачивания итогового PDF-отчёта.

## Что умеет frontend

- вход и регистрация поверх cookie-based auth backend;
- обзорная панель по состоянию пайплайна;
- скачивание итогового PDF-отчёта прямо с обзорной панели;
- выделенный блок `AQI` на обзоре с последней точкой прогноза и категорией риска;
- интерактивная карта воздуха с временной шкалой, городским фоном и hot spots;
- сбор наблюдений и просмотр последних записей;
- русскоязычные названия метрик и семантическая подсветка риска;
- сборка `DatasetSnapshot`;
- обучение и просмотр версий моделей;
- генерация прогнозов, `backtest` и оценка по факту;
- запуск экспериментальных прогонов и серий;
- контроль фоновых и отложенных задач в человекочитаемом виде;
- скачиваемая инструкция по интерфейсу прямо из шапки приложения.

## Экранная модель

- `/login` — вход и регистрация.
- `/app` — обзор пайплайна.
- `/app/air-map` — карта качества воздуха по постам и городскому фону.
- `/app/observations` — сбор наблюдений и фильтры по последним данным.
- `/app/datasets` — сборка и просмотр срезов датасета.
- `/app/models` — обучение, активная модель и лидерборд.
- `/app/forecasts` — прогнозы, ретропроверка и оценка.
- `/app/experiments` — исследовательские прогоны и серии.
- `/app/tasks` — мониторинг текущих и отложенных задач.

## Что важно в текущем поведении интерфейса

- интерфейс старается подставлять параметры последнего датасета, активной модели и последнего прогноза;
- в списке прогнозов показываются только модели со статусом `ready`;
- кнопка оценки по факту активируется только после появления наблюдений на весь горизонт прогноза;
- задачи хранятся в `localStorage`, чтобы не теряться между переходами;
- любую фоновую задачу можно добавить в мониторинг вручную по `task_id`;
- overview использует отдельный агрегированный backend endpoint, а не длину урезанных списков;
- карта использует `OpenFreeMap + MapLibre`, backend snapshot `/api/monitoring/air-map` и временную шкалу `4 дня назад + 1 день вперёд`;
- страница задач теперь показывает карточки с человеческими статусами, краткими результатами и скрытыми техническими деталями вместо стены сырого JSON;
- итоговый PDF-отчёт скачивается прямо из интерфейса и оформлен как финальная аналитическая сводка.

## Требования

- Node.js 20+
- npm 10+
- запущенный backend `air_monitor_back`

Для разработки предполагается backend по адресу:

```text
http://127.0.0.1:8000
```

## Конфигурация

Основные переменные окружения:

- `VITE_BACKEND_TARGET` — адрес backend для Vite proxy в dev-режиме;
- `VITE_API_BASE_URL` — базовый URL API для production-сборки;
- `VITE_AIR_MAP_STYLE_URL` — URL style для карты воздуха, по умолчанию используется `OpenFreeMap Liberty`.

Быстрый старт:

```bash
cp .env.example .env
```

## Локальный запуск

## Docker deploy

Frontend deploy-контур собирает статический билд и отдаёт его через `nginx`, одновременно проксируя API-запросы на backend внутри общей Docker-сети.

Основные команды:

```bash
make up
make build
make ps
make logs
make down
```

Если менялся код интерфейса, одного `make up` недостаточно: сначала нужен `make build` или `make rebuild`, потом уже `make up`.

Если backend внутри сети доступен под другим именем или портом:

```bash
make up BACKEND_UPSTREAM_HOST=custom-backend BACKEND_UPSTREAM_PORT=8000 FRONTEND_PORT=8080
```

Если нужно поднять оба репозитория одной командой, запуск делается из backend:

```bash
cd /path/to/air_monitor_back
make deploy-up-all FRONTEND_DIR=/path/to/air_monitor_front
```

## Пользовательская инструкция

В шапке приложения рядом с блоком пользователя есть кнопка `Инструкция`. Она скачивает markdown-гайд по интерфейсу с описанием экранов, полей форм и рабочего сценария.

Исходный файл лежит в:

```text
public/air-monitor-user-guide.md
```

Этот файл нужно обновлять одновременно с изменением экранов, потому что именно он скачивается пользователем в production.

## Безопасность и зависимости

- фронт собран на `Vite 6.4.2+`, потому что `6.4.1` помечался уязвимым в advisory;
- при обновлении зависимостей стоит перечитывать `package-lock.json` и переустанавливать `node_modules`, если IDE продолжает показывать старые предупреждения.

## Связанный backend

Проект работает вместе с backend-репозиторием `air_monitor_back`.

## Статус

Это прикладной интерфейс под конкретный backend и исследовательский pipeline. Проект сознательно не пытается быть универсальным dashboard-конструктором и отражает предметную модель backend почти без абстракций.
