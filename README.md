# air_monitor_front

Frontend магистерского проекта по мониторингу и краткосрочному прогнозированию качества воздуха в Норильском промышленном районе.

Проект предоставляет пользовательский веб-интерфейс для существующего backend API: аутентификация, сбор наблюдений, подготовка датасетов, обучение моделей, построение прогнозов, ретропроверка, оценка качества и работа с сериями экспериментов.

## Назначение проекта

Система решает две задачи:

1. Прикладную.
   Дать удобный интерфейс для работы с ML-пайплайном без ручного использования Swagger, Celery и прямых API-запросов.

2. Презентационную и исследовательскую.
   Предоставить понятный веб-интерфейс для демонстрации магистерской работы: от получения данных до анализа результатов моделей и экспериментов.

## Текущий функционал

- логин и регистрация поверх cookie-based auth backend;
- обзорная панель по состоянию пайплайна;
- просмотр наблюдений и запуск сбора данных;
- сборка срезов датасета;
- обучение моделей и просмотр активной модели;
- построение прогнозов;
- ретропроверка от исторической точки отсечения;
- оценка прогноза по фактическим наблюдениям;
- запуск и просмотр экспериментов и серий экспериментов;
- мониторинг фоновых и отложенных задач;
- автоподстановка разумных параметров из backend там, где это возможно.

## Архитектура

### Структура `src`

- `app`
  Корневая сборка приложения, роутинг, провайдеры.
- `components`
  Общие UI-компоненты и оболочка приложения.
- `hooks`
  Клиентская логика аутентификации и локального трекинга задач.
- `lib`
  API-клиент, форматирование и вспомогательные утилиты.
- `pages`
  Экранные компоненты по предметным разделам.
- `types`
  Типы и контракты API.

### Принципы структуры

- предметные экраны разнесены по отдельным страницам;
- интеграция с backend сосредоточена в одном API-слое;
- общие элементы интерфейса вынесены в переиспользуемые UI-компоненты;
- сетевые данные и их кэширование управляются через TanStack Query;
- стили собраны на Tailwind с небольшим набором собственных UI-классов.

## Экранная структура

### `/login`

Авторизация и регистрация пользователя.

### `/app`

Обзор пайплайна:

- наблюдения;
- датасеты;
- модели;
- прогнозы;
- эксперименты;
- серии.

### `/app/observations`

Сбор и просмотр наблюдений.

### `/app/datasets`

Сборка и просмотр срезов датасета.

### `/app/models`

Обучение, просмотр версий и сравнение метрик моделей.

### `/app/forecasts`

Построение прогноза, ретропроверка и оценка.

### `/app/experiments`

Создание серий и запуск исследовательских прогонов.

### `/app/tasks`

Мониторинг фоновых и отложенных задач.

## Требования

- Node.js 20+
- npm 10+
- запущенный backend `air_monitor_back`

Для локальной разработки предполагается, что backend доступен по:

```text
http://127.0.0.1:8000
```

## Конфигурация

Проект использует две переменные окружения:

- `VITE_BACKEND_TARGET`
  Адрес backend для Vite proxy в dev-режиме.
- `VITE_API_BASE_URL`
  Базовый URL API для production-сборки или нестандартного окружения.

Пример:

```bash
cp .env.example .env
```

## Локальный запуск

1. Установить зависимости:

```bash
npm install
```

2. Убедиться, что backend поднят и отвечает.

3. Запустить dev-сервер:

```bash
npm run dev
```

По умолчанию приложение будет доступно по адресу:

```text
http://127.0.0.1:5173
```

Если backend слушает другой адрес, можно запустить так:

```bash
VITE_BACKEND_TARGET=http://127.0.0.1:8001 npm run dev
```

## Интеграция с backend

В dev-режиме запросы `/api` и `/health` проксируются через Vite на backend.

Это даёт:

- работу с `HttpOnly` cookie без отдельного CORS-конфига для localhost;
- единый origin для фронта и проксируемого API;
- простой запуск рядом с Docker-контуром backend.

API-клиент реализован в `src/lib/api.ts`.

## Что важно для демонстрации

Основной сценарий использования:

1. Собрать наблюдения.
2. Построить срез датасета.
3. Обучить модель.
4. Построить прогноз.
5. При необходимости выполнить ретропроверку и оценку.
6. Запустить один или несколько экспериментов.

Фронт старается автоматически подставлять параметры из backend:

- активную модель;
- последний срез датасета;
- последнюю точку отсечения прогноза для ретропроверки.
